## What we did
This session had two phases. Phase 1 (earlier, from context summary): migrated the site from plain HTML to Astro 5, connected Sanity CMS, seeded all 6 weekly event cards with real content, set up Sanity → Vercel webhook for auto-rebuild on publish, and started the Sanity Presentation tool (live preview) setup. Phase 2 (this session): debugged and fixed the Presentation tool "Unable to connect" error through multiple rounds — root causes were a CSP header conflict, a wrong import path, and a comlink version mismatch between the Studio and the site.

## Decisions made
- **React added to Astro** (`@astrojs/react`): Required by `@sanity/visual-editing`. Does not affect the site's visual output — no React components are rendered.
- **`@sanity/client` upgraded to v7**: Required peer dep for `@sanity/visual-editing`.
- **`@sanity/visual-editing` pinned to v2.x (not v5.x)**: v5.x uses `presentation-comlink@2` but the Studio's `sanity@3.99.0` uses comlink v1. Mismatch caused "Unable to connect." v2.15.4 uses comlink v1 — both sides match.
- **CSP `frame-ancestors` removed from vercel.json**: Adding it caused the iframe block. Public restaurant site — no need to restrict framing.
- **`sanity/presentation` import (not `@sanity/presentation`)**: The standalone `@sanity/presentation` package is deprecated as of v2.0.0. It now lives inside `sanity/presentation`.
- **Stega encoding on `@sanity/client/stega`**: Enabled in `src/lib/sanity.js`. Embeds invisible field metadata in fetched strings so the overlay knows which element maps to which Sanity field.
- **Studio redeployed**: `npx sanity deploy` run after fixing the import — Studio at harpers-brewpub.sanity.studio is current.
