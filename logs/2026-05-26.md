## What we did
Continued from 2026-05-21 session. Finished wiring up Sanity visual editing and fixed two bugs introduced by enabling stega encoding. Also confirmed mobile and desktop card styling is visually consistent.

## Bugs found and fixed

### Bug 1 — Stega encoding corrupted CSS class names
Enabling `@sanity/client/stega` embeds invisible Unicode characters into ALL strings returned from Sanity — including `event.day`. That field is used to generate CSS class names like `card-monday`. The invisible chars broke the selector match, so no card had a background image (desktop and mobile both affected).

Fix: imported `stegaClean` from `@sanity/client/stega` and wrapped the class name use:
```js
`card-${stegaClean(event.day).toLowerCase()}`
```
File changed: `src/pages/index.astro`

### Bug 2 — Visual editing overlay showing on public site
`enableVisualEditing()` was called unconditionally, so every visitor saw the Sanity "Open in Studio" overlays on the event cards on the live public website.

Fix: wrapped the call in an iframe check — overlays only activate when the page is loaded inside the Sanity Presentation tool:
```js
if (window !== window.parent) {
  enableVisualEditing({ zIndex: 1000 });
}
```
File changed: `src/layouts/BaseLayout.astro`

## Visual parity check (mobile vs desktop)
Confirmed: mobile and desktop non-hover card state are identical — same dark gradient overlay, photo behind. Desktop hover reveals the full photo (overlay fades). Mobile has no hover, so stays in the default state. This is intentional and correct — nothing to change.
